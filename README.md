## ⚠️ INSTRUCTIONS FOR CLAUDE (AI AGENT)

> This section is addressed directly to Claude or any AI reading this file. If you are a human, skip to [How To Use](#how-to-use).

You are the **Vibe Coding Pro Planning Assistant**. When a user pastes this framework link alongside their app idea, your job is to:

1. **Interview** them using the Question Framework below
2. **Generate** a structured `.txt` planning document in the exact Output Format specified
3. **Recommend** the right components from the JB Component Registry for their project
4. **Structure** their build into clear phases with tasks and ready-to-paste Lovable/V0 prompts

### Your Behaviour Rules

- Ask **one question at a time** unless grouping 2–3 tightly related questions
- Never suggest a local development setup — this framework is for **Lovable and V0 only** (no terminal, no local Node.js)
- Always recommend JB Registry components **before** suggesting AI write anything from scratch
- Every phase must end with a **ready-to-paste prompt** for Lovable or V0
- When generating prompts for Lovable, always include the relevant JB component install command so Lovable reads the component first before building
- Keep phases **small and focused** — one phase = one concern (auth, data, UI, payments, deploy)
- The final document must be **copy-paste ready** — no placeholders left unfilled

## What Is This Framework?

The Vibe Coding Pro Framework is a **planning + prompting system** that solves the 6 most common reasons AI-built apps fail:

| Problem | Framework Solution |
|---|---|
| Usage limits spike to $100–$200 per project | JB components cut token usage 60–80% |
| AI generates insecure authentication | Better Auth UI — one command, battle-tested |
| Every app looks the same (purple gradient slop) | Design system prompts + Website UI component |
| AI gets stuck in endless fix loops | Phase decomposition + Rescue Prompt templates |
| Inconsistent UI with no reusability | Shared component library across all projects |
| Deployment confusion and domain pain | Vercel + Cloudflare step-by-step phase |

## The Phase Framework

Every project is broken into **5 standard phases**. Depending on the project type, some phases may be skipped or merged.

### Phase 1 — Foundation
**Goal:** Project is created, design system is locked, database is connected, and authentication works.  
**Standard Tasks:**
- Create project in Lovable (or V0 for UI-first)
- Define and apply the design system prompt
- Install Website UI or a custom layout foundation
- Install Better Auth UI (always — do not skip)
- Connect Neon database and run Prisma migration
- Verify login, signup, Google OAuth, and protected routes work

### Phase 2 — Core Features
**Goal:** The main value-delivery screens of the app are built and connected to real data.  
**Standard Tasks:**
- Install relevant JB data/form/display components
- Build primary CRUD screens (lists, create, edit, delete)
- Connect all screens to the Neon database via Prisma
- Add file uploads if required (File Storage UI)
- Ensure all pages respect authentication state

### Phase 3 — Payments & Billing (skip if no monetisation)
**Goal:** Users can pay, subscriptions are tracked, and features are gated.  
**Standard Tasks:**
- Install Stripe UI component
- Configure products and pricing in Stripe dashboard
- Set up Stripe webhooks
- Gate premium features behind subscription status
- Add billing management page (upgrade, cancel, invoice)

### Phase 4 — Email & Notifications (skip if no emails needed)
**Goal:** App communicates with users automatically via email.  
**Standard Tasks:**
- Connect Resend and verify sending domain
- Build welcome email (on signup)
- Build password reset email (wired to Better Auth)
- Build payment receipt email (triggered by Stripe webhook)
- Install Consent Manager for GDPR compliance

### Phase 5 — Deployment & Go Live
**Goal:** App is live on a custom domain, all environment variables are set, and emails land in inbox.  
**Standard Tasks:**
- Deploy to Vercel
- Set all environment variables in Vercel dashboard
- Purchase domain and configure Cloudflare DNS
- Connect custom domain to Vercel (SSL auto-included)
- Verify email sending domain in Resend (for inbox delivery)
- Run production checklist (see below)

### Production Checklist (Phase 5)
- [ ] Auth flows tested on mobile and desktop
- [ ] Stripe payment processed successfully in test mode
- [ ] All environment variables set in Vercel
- [ ] Database migrations applied to production Neon
- [ ] Custom domain live with green SSL padlock
- [ ] Password reset email received and link works
- [ ] Welcome email hits inbox (not spam)
- [ ] File uploads tested in production
- [ ] All protected routes redirect to login when logged out
- [ ] 404 and error pages are styled and branded
- [ ] Consent banner visible on first visit

## Question Framework (For Claude)

When a user pastes their idea, ask questions from these categories. Not all questions apply to every project — use judgment.

### Category A — Core Understanding
1. What is the main problem this app solves? Who has that problem?
2. Who are the users? (single user, multiple users, admin + users, public + private)
3. Is this for personal use, client delivery, or commercial launch?

### Category B — Features & Scope
4. What are the 3 most important things a user must be able to do in this app?
5. Does the app need user accounts and login? (almost always yes — but confirm)
6. Does the app need different user roles? (e.g. admin vs regular user)
7. Does the app need to store and display data in tables or lists?
8. Does the app need file uploads? (images, documents, attachments)

### Category C — Monetisation
9. Will users pay for this app? (one-time, subscription, or free)
10. If subscription: what features are free vs paid?

### Category D — Communication
11. Does the app need to send emails? (welcome, notifications, receipts)
12. Does the app need any other notifications? (in-app, SMS — note: SMS is out of scope for this framework)

### Category E — Design & Brand
13. Do you have a brand colour in mind, or should the framework suggest one based on the app's tone?
14. What 3 words describe how the app should feel? (e.g. "professional, clean, fast" or "friendly, colourful, playful")
15. Are there any apps whose design you admire that we could draw inspiration from?

### Category F — Timeline & Constraints
16. Is there a deadline for this build?
17. Are there any features that are strictly out of scope for this version?

## How To Use

### Step 1 — Copy the user prompt
Copy the contents of [`CLAUDE_PROMPT.md`](./CLAUDE_PROMPT.md) from this repository.

### Step 2 — Open Claude Web
Go to [claude.ai](https://claude.ai) and start a new conversation.

### Step 3 — Paste and add your idea
Paste the contents of `CLAUDE_PROMPT.md` into the Claude message box, then add a line break and write your app idea in plain English. Example:

```
[CLAUDE_PROMPT.md contents pasted here]

MY IDEA: I want to build a school management system where teachers can manage students, track attendance, and parents can log in to see their child's progress and pay school fees.
```

### Step 4 — Answer Claude's questions
Claude will ask you 6–10 questions about your project. Answer them honestly and completely.

### Step 5 — Get your planning document
Claude will generate a complete `.txt` planning document. Copy it and save it — this is your build blueprint.

### Step 6 — Start building
Open Lovable or V0. Paste **Phase 1's Lovable Prompt** as your very first message. Follow the phases in order.

## Design System Prompt Template

Use this at the **start of every new Lovable session** to lock your design and prevent AI slop:

```
DESIGN SYSTEM — apply this to every component you create or modify:

Primary color: [YOUR HEX]
Background: white (#FFFFFF) for light mode, #0F172A for dark mode
Font: Inter (already included via Tailwind)
Border radius: rounded-xl for cards, rounded-lg for buttons and inputs
Shadows: shadow-sm only — no dramatic shadows
Cards: bg-white border border-slate-200 shadow-sm
Buttons: bg-[primary] text-white hover:opacity-90 rounded-lg px-4 py-2
Inputs: border border-slate-300 rounded-lg focus:ring-2 focus:ring-[primary] focus:border-transparent
Typography: text-slate-900 headings, text-slate-600 body, text-slate-400 muted

RULES:
- No purple gradients
- No neon colors
- No heavy shadows or dramatic effects
- Every page should feel like a premium B2B SaaS product (think Linear, Notion, Vercel dashboard)
- Consistent spacing: use Tailwind spacing scale (4, 6, 8, 12, 16, 24)
- Mobile-first responsive design on all pages
```

## Lovable vs V0 Decision Framework

| Use V0 when... | Use Lovable when... |
|---|---|
| You want to design a landing page | You want to connect a page to a database |
| You want a specific UI component | You want to wire up authentication |
| The design needs to look unique | You want to build an API route |
| You are iterating visually | You want to handle form submissions |
| You want to try 3 layout options fast | You want to deploy the full application |

**The Pro Workflow:**
1. Design the component or page in V0
2. Copy the generated code
3. In Lovable, say: *"Add this component to the project without changing any of the design. Only wire up the functionality: [paste V0 code]"*
